MongoDB在线修改oplogSize

2016-01-28
  • oplog简介
  • 修改oplogSize大小
  • 回滚

oplog简介

oplog（operations log）是一个特殊的固定集合（capped collection），用来记录所有的“写”（insert、update、delete以及修改索引等）操作纪录。MongoDB将数据库操作应用到primary然后记录到primary的oplog。secondary从primary复制oplog并将操作应用到自己（异步，有一定延迟，如果oplog大小不当，可能出现oplog溢出，从节点不可用）。所有的副本集成员维持一个oplog的副本来维持数据库的当前状态。
对于64位的Linux, Solaris, FreeBSD和Windows, MongoDB默认分配可用磁盘空间的5%作为oplog。如果这个值小于1G，则分配1G作为oplog。
oplog是一个固定集合（capped collection）,所以在正常操作期间不能修改他的大小。在集群初始化的时候最好考虑好该值的大小，因为后续修改起来比较麻烦。
在大多数情况下默认的oplog大小是可接受的。Oplog的大小由应用的操作行为决定。如果应用经常在短时间内执行大量的批量update、delete操作，从而产生大量的“写”操作纪录，则应该适当增大oplog的大小，避免primary的旧的oplog记录在被secondary同步走之前被新产生的oplog纪录覆盖掉。

修改oplogSize大小

注意：尽量选择集群较空闲的时间段进行，减少单节点的操作时间，确保重启后的节点能够跟得上集群的oplog。最好先操作所有的secondary节点再操作primary节点。操作primary时，先使用stepdown降级，待集群选举出新的primary后再shutdown。

对副本集中所有的节点执行以下操作：

  1. 查看当前oplogsize大小

  2. 关闭当前节点，登陆其他节点查看集群状态是否正常

  3. 以standalone的方式用另外的端口（确保端口空闲）启动

  4. 备份当前的oplog

  5. 连接mongd

  6. 保存oplog的最后一条纪录

  7. 删除oplog.rs

  8. 创建新的oplog，并指定大小，如1073741824 = 1GB

  9. 插入之前保存的旧oplog的最后一条纪录

  10. 关闭该节点

  11. 修改配置文件中的oplogSize大小（不是必须）

  12. 以副本集的形式重启该节点

  13. 查看oplogsize大小和集群状态

回滚

使用mongorestore恢复oplog.rs

MongoDB OPS Manager介绍

1. 简介

MongoDB OPS Manager是随MongoDB Enterprise版本发布的MongoDB管理、监控、备份系统(但是也可以用于community版本)，是MMS(MongoDB Management Service)的内网版本。

MongoDB Management Service (MMS) is a suite of services for managing MongoDB deployments. Engineered by the team who develops MongoDB, MMS makes it easier to operate MongoDB at any scale. MMS provides monitoring, backup and recovery, helping users optimize clusters and mitigate operational risk. MMS users can visualize database performance and set custom alerts that notify when particular metrics are out of normal range. MMS is also the only continuous backup solution for MongoDB, providing point-in-time recovery for replica sets and cluster-wide snapshots of sharded systems.
MMS is available as a fully managed cloud service, or as on-prem software included with your MongoDB Enterprise subscription.

MMS工作原理如下：

  1. 在MMS服务器上配置你的MongoDB信息（Host，Port，User，Passwd等）
  2. 在一台能够访问你MongoDB服务的内网机器上运行其提供的Agent
  3. Agent脚本从MMS服务器获取到你配置的MongoDB信息
  4. Agent脚本连接到相应的MongoDB获取必要的监控数据
  5. Agent脚本将监控数据上传到MMS的服务器
  6. 登录MMS网站查看整理过后的监控数据图表了

MMS是收费的！！！！OPS可以自己在内网部署使用。

OPS主要功能：

  1. 监控：实时的数据可视化展示，关键数据库指标监控报警
  2. 管理：自动部署，修改配置，启停等
  3. 备份：数据备份、恢复，支持scheduled snapshots and point-in-time recovery

本文档只针对管理和部署功能进行实际部署试用。

Automation Agent用于管理、部署mongodb以及其他agent，需要首先进行安装。按照指示进行安装。

MongoDB离线创建索引

2016-01-28
  • 目的
  • 原理
  • 步骤
    • 1. 部署新的hidden节点
    • 2. 添加hidden节点
    • 3. 等待hidden节点完成initial sync（此方法耗时很长，但可以获得一份压缩后的数据）
    • 4. 检查集群状态，确认一切正常，继续
    • 5. 关闭hidden节点，检查集群状态，确认一切正常，继续
    • 6. 以standalone方式启动hidden节点
    • 7. 登陆hidden节点，执行创建索引动作
    • 8. 待创建完毕，关闭hidden节点，以副本集方式启动
    • 9. 检查集群状态和索引状态，确认一切正常，继续
    • 10. 将hidden节点的新数据，逐个拷贝到集群中的其他secondary节点，并使用新数据重启mongod。对于primary节点，先使用rs.stepdown()，待新的primary节点产生后再进行重启。
    • 11. 移除hidden节点
    • 12. 清理线上关于hidden节点的hosts、iptables配置
  • 回滚

目的

阐述一种风险较低，耗时较少的MongodDB创建大量索引的方案。
适用于 MongodDB2.x副本集。

原理

MongodDB2.2前台创建索引命令会阻塞实例的复制和读写操作，虽然可以通过指定background选项来避免，但是从节点依然是前台创建索引，而且在大数据集上建立索引耗时很长。
官方建议将副本集的节点摘下之后进行创建索引操作，然后再加入集群。
通过initial sync或者复制集群其他节点的数据在副本集中临时添加hidden节点，摘下hidden节点创建索引，然后将hidden节点的数据依次拷贝到集群中的其他节点，实现整个集群的索引创建。

参考资料：
https://docs.mongodb.org/v2.2/core/indexes/#index-creation-background
https://docs.mongodb.org/v2.2/administration/indexes/#index-building-replica-sets

步骤

1. 部署新的hidden节点

  • 配置hosts、iptables，保证新节点可以顺利与集群其他节点通信
  • 配置mongodb并启动，配置文件参考集群其他节点

2. 添加hidden节点

  • 登陆到集群的primary，执行

  • 检查集群状态

3. 等待hidden节点完成initial sync（此方法耗时很长，但可以获得一份压缩后的数据）

4. 检查集群状态，确认一切正常，继续

5. 关闭hidden节点，检查集群状态，确认一切正常，继续

6. 以standalone方式启动hidden节点

7. 登陆hidden节点，执行创建索引动作

8. 待创建完毕，关闭hidden节点，以副本集方式启动

9. 检查集群状态和索引状态，确认一切正常，继续

10. 将hidden节点的新数据，逐个拷贝到集群中的其他secondary节点，并使用新数据重启mongod。对于primary节点，先使用rs.stepdown()，待新的primary节点产生后再进行重启。

11. 移除hidden节点

  • 关闭hidden节点的mongod
  • 登陆primary节点执行

12. 清理线上关于hidden节点的hosts、iptables配置

至此，完成整个集群的索引创建

回滚

由于原始数据会进行备份，所以回滚只需替换回原来的数据启动。

MongoDB硬盘空间紧缩

2016-01-28
  • 目的
  • 原理
  • 步骤
    • 1. 部署新的hidden节点
    • 2. 添加hidden节点
    • 3. 等待hidden节点完成initial sync
    • 4. 检查集群状态，确认一切正常，继续
    • 5. 将hidden节点的新数据，逐个拷贝到集群中的其他secondary节点，并使用新的数据文件启动mongod（原始数据文件进行备份）。对于primary节点，先使用rs.stepdown()，待新的primary节点产生后再进行操作。
    • 6. 移除hidden节点
    • 7. 清理线上关于hidden节点的hosts、iptables配置
  • 回滚

目的

阐述一种风险较低，耗时较少的MongoDB硬盘空间紧缩方案。
适用于MongoDB2.x副本集。

原理

MongoDB2.2只有在dropdatabase时才会释放硬盘空间，删除document和collection不会释放硬盘空间，而是重复利用。而且随着数据库数据的增、删、改，会产生大量碎片，导致100G的数据可能占用300G的空间，数据紧缩的需求应运而生。
数据紧缩实际是通过MongoDB的initial sync数据重建来实现的。
通过在副本集中临时添加hidden节点，通过hidden节点的initial sync获得一份紧缩后的全量数据，然后将这份数据依次拷贝到集群中的其他节点，实现整个集群的硬盘空间紧缩。

参考资料：
https://docs.mongodb.org/v2.2/administration/replica-sets/#resyncing-a-member-of-a-replica-set

步骤

1. 部署新的hidden节点

  • 配置hosts、iptables，保证新节点可以顺利与集群其他节点通信
  • 配置mongodb并启动，配置文件参考集群其他节点

    2. 添加hidden节点

  • 登陆到集群的primary，执行

  • 检查集群状态

3. 等待hidden节点完成initial sync

4. 检查集群状态，确认一切正常，继续

5. 将hidden节点的新数据，逐个拷贝到集群中的其他secondary节点，并使用新的数据文件启动mongod（原始数据文件进行备份）。对于primary节点，先使用rs.stepdown()，待新的primary节点产生后再进行操作。

6. 移除hidden节点

  • 关闭hidden节点的mongod
  • 登陆primary节点执行

7. 清理线上关于hidden节点的hosts、iptables配置

至此，完成整个集群的硬盘空间压缩

回滚

由于原始数据会进行备份，所以回滚只需替换回原来的数据启动。

Hadoop监控方案总结

2016-01-22

本文结合自己在Hadoop运维工作中的经验，对Hadoop的监控指标和可用的监控手段进行总结。

  • 一、 监控类别
    • 1. 基础监控
    • 2. 应用监控
    • 3. 监控图形展现
  • 二、 可用的监控方法
    • 1. ganglia
    • 2. zabbix
    • 3. 网页分析（结合第三方监控系统）
    • 4. jmx（结合第三方监控系统）
    • 5. Ganglia数据源（结合第三方监控系统）
    • 6. hadoop cmd（结合第三方监控系统）

一、 监控类别

1. 基础监控

（1）机器可用性监控（单机粒度，针对Hadoop集群的所有机器）

  • 机器存活PING
  • 远程登录SSH

建议报警方式：短信
（2）机器资源占用监控（单机粒度，针对Master；集群粒度，针对所有Slave）

  • CPU. % CPU Utilization and periodic (1, 5, 15 minutes) CPU load averages
  • DISK. Average Disk data transfer rates (I/O bandwidth) and Number of disk I/O operations per second
  • DISK USAGE(%)
  • NETWORK. Average data transfer rate and network latency
  • MEMORY. Average Memory and Swap space utilization

建议报警方式：根据阈值进行邮件或短信报警。

2. 应用监控

（1）存活类：进程监控（单机粒度）
监控namenode、datanode等程序的进程是否存在。

建议报警方式：主节点短信，从节点邮件
（2）存活类：端口监控（单机粒度）

建议报警方式：主节点短信，从节点邮件
（3）可用性监控

  • HDFS Safe Mode Status
  • NameNode edit logs directory status
  • 通过执行有代表性的操作确保服务响应正常。如创建目录、上传文件、下载文件、删除文件、执行m/r等

建议报警方式：短信
（4）容量监控

  • HDFS capacity utilization
  • HDFS blocks utilization

建议报警方式：根据阈值进行邮件或短信报警
（5）集群节点状态监控

  • dataNodes (live, dead, and decom)
  • nodeManager (active, lost, and decom)
  • regionServers (live, dead, in transition)

建议报警方式：根据阈值进行邮件或短信报警
（6）数据完整性监控
Monitor nodes with potential of corrupted data (number of under-replicated blocks).
建议报警方式：邮件
（7）访问延时监控

  • NameNode RPC latency alert. This alert is triggered if the NameNode operations (RPC) latency exceeds the configured critical threshold. Typically an increase in the RPC processing time increases the RPC queue length, causing the average queue wait time to increase for NameNode operations.
  • HBase RegionServers average read latency. the HDFS read latency for the Region Servers aggregated across all the region servers (slaves)
  • HBase RegionServers average write latency. the HDFS write latency for the Region Servers aggregated across all the region servers (slaves)
  • HBase RegionServers RPC Average Waiting Time. the average wait time aggregated across all the region servers (slaves)

建议报警方式：邮件
（8）日志监控
对每个服务的错误日志进行监控，日志格式通过log4j配置，邮件报警。
建议报警方式：邮件

3. 监控图形展现

通过图表方式对整个集群的CPU、内存、磁盘、网络、I/O等信息进行展现。

二、 可用的监控方法

1. ganglia

能够采集系统级别和Hadoop集群的度量信息，支持聚合。

  • 通过gmond获取机器的CPU、内存、磁盘等系统相关信息。
  • Hadoop原生支持ganglia（hadoop-metrics2.properties，hadoop-metrics2-hbase.properties），直接通过nodemanager、datanode、regionserver等节点发送的数据获取hadoop自身的度量信息。

不支持报警邮件和短信，解决方案：ganglia-alert、gangila + nagios

2. zabbix

采集系统级别、hadoop本身的度量信息（需要插件Apache Hadoop Plugin、Apache HBase Plugin）
不支持聚合

3. 网页分析（结合第三方监控系统）

通过访问以下页面，获取各节点信息，以插件方式集成到第三方监控该系统中。
http://master:50030 查看MapReduce上的resourcemanager
http://master:50060 查看MapReduce上的node manager
http://master:50070 查看HDFS上的节点信息namenode
http://master:60010/master.jsp 查看master节点信息 hmaster
http://master:60030/regionserver.jsp 查看regionserver信息
http://master:60010/zk.jsp 查看zookeeper信息

4. jmx（结合第三方监控系统）

Java Management Extensions (JMX)，Hadoop自带的监控接口。
NameNode的信息：http://namenode:50070/jmx
DataNode的信息：http://datanode:50075/jmx
http://your_namenode:50070/jmx?qry=Hadoop:service=NameNode,name=NameNodeInfo
返回值全部是JSON，非常便于自己进行处理。返回的信息（hadoop+jvm）也非常详细，内存状态，java堆信息等等。甚至还有操作系统信息，版本，JVM版本信息等等，很全面。

5. Ganglia数据源（结合第三方监控系统）

可以直接调用gmeta接口获取ganglia所采集的Hadoop集群的度量数据
curl ‘http://127.0.0.1:8652/hadoop?filter=summary'
curl ‘http://127.0.0.1:8652/hadoop/yf3493.sogou-in.domain/disk_free'

Ganglia API
The Ganglia API is a small standalone python application that takes XML data from a number of Ganglia gmetad processes and presents a RESTful JSON API of the most recently received data.

6. hadoop cmd（结合第三方监控系统）

hadoop dfsadmin -report
用这个命令可以快速定位出哪些节点down掉了，HDFS的容量以及使用了多少，以及每个节点的硬盘使用情况。当然NameNode有个http页面也可以查询，但是这个命令的输出更适合我们的脚本监控dfs的使用状况
hadoop fsck /
用这个命令可以检查整个文件系统的健康状况，但是要注意它不会主动恢复备份缺失的block，这个是由NameNode单独的线程异步处理的。

https://github.com/winway/a-simple-monitor-framework/tree/v2.0

  • 1. 简介
  • 2. 配置

1. 简介

程序包括以下文件和文件夹

  • monitor.sh
    主程序，会依次调用snatch.sh、analyse.sh、format.sh、verify.sh、display.sh
  • snatch.sh
    数据采集，基于mussh（不了解的可以参考下这里）实现，周期性抓取各服务器上感兴趣的数据项，存储到data文件夹中
  • analyse.sh
    数据分析，根据etc文件夹中的阈值配置文件，产生需要告警的条目，主要工作由raw_analyse.py完成
  • format.sh
    数据格式化，将原始的纯文本数据格式化为html格式，主要工作由raw2html.py完成
  • verify.sh
    数据完整性检查，将抓取的数据与配置文件进行对比，确定有误漏掉的监控对象
  • display.sh
    数据展示，更新display文件夹下的软连接，指向最新的html文件
  • time_sync.sh
    时间同步，由于某种原因，无法使用ntp，所以使用脚本设置被监控服务器的时间
  • data_clear.sh
    数据清理，清除data目录下的旧文件
  • config.sh
    一些变量的定义
